## 本课目标

- [ ] 能解释 PDM 如何只用 1 个 bit 编码连续的声音幅度
- [ ] 能说清过采样 + 抽取滤波(decimation)为什么能"无中生有"出 16-bit 精度
- [ ] 能对比麦克风用 PDM、扬声器用 I2S 各自的理由

## 硬件原理

上一课你往外"吐"声音:16-bit PCM 样本走 I2S 三线进 NS4168。这一课反过来,从麦克风往里"吸"。但 Cardputer 上的 SPM1423 MEMS 麦克风给你的,不是你熟悉的 16-bit 样本——它吐出的是一条 **每个时钟只有 1 个 bit** 的高速脉冲流。这就是 PDM(Pulse Density Modulation,脉冲密度调制)。

**1 个 bit 怎么可能表达音量?** 类比调光:LED 只有开/关两个状态,但你让它每秒闪几十万次,亮的比例高就显得亮,比例低就显得暗——人眼是天然的低通滤波器,把高速开关"糊"成了连续亮度。PDM 一模一样:1 的密度高 = 波形此刻幅度高,1 的密度低 = 幅度低。

```
模拟波形:    ／￣￣＼            ＿＿
            ／      ＼＿＿＿＿＿／
PDM 流:   1101111011 1010 0100010 0101101
           ↑密度高=波峰   ↑密度低=波谷
```

代价是速度:想用密度表达细腻的幅度,时钟必须远高于音频频率——PDM 时钟通常在 1~3 MHz,是 16 kHz 采样率的一百倍上下。这叫**过采样**。麦克风内部的 sigma-delta 调制器还有个聪明设计:它把量化噪声"推"到高频段,而人声都在低频段。

于是接收端要做的事就明确了:**抽取滤波(decimation)**——低通滤掉高频噪声,再把每一大段(比如 64 个)1-bit 脉冲"浓缩"成一个 16-bit PCM 样本。直觉上就是"数一数这 64 个 bit 里有几个 1"(实际是更精巧的 CIC/FIR 滤波)。3.072 MHz 的 1-bit 流 ÷ 64 = 48 kHz 的 16-bit 流,精度是拿速度换来的。好消息:ESP32-S3 的 I2S 外设有硬件 PDM 接收模式,这套滤波它包了,DMA 直接往内存里送现成的 PCM——和第 5 课的 DMA 是同一个思想,方向相反。

**那为什么麦克风偏爱 PDM、扬声器侧却走 I2S?** 看谁负担得起复杂度。麦克风要做得米粒一样小、几毛钱一颗,PDM 只需时钟+数据两根线,发送端几乎零逻辑,把滤波的苦活全甩给 SoC;而且两颗 PDM 麦可以共享一根数据线(各占时钟的上升/下降沿),立体声不加线。I2S 传的是现成 PCM,接收方(功放)简单,但要三根线、发送方逻辑更重。一句话:**PDM 让麦克风傻、主控累;I2S 让数据规整、连线多**。各在各的位置上最优。

最后一块拼图是软件:声音是持续的河流,屏幕一次只画一帧。所以要用**采集-处理-绘制流水线**——DMA 在后台默默灌缓冲,你的代码周期性地取一段、算一算、画上去。取的速度必须跟上灌的速度,否则数据被覆盖,波形出现断裂。这正是环形缓冲(ring buffer)解决的问题:写指针(DMA)在前面绕圈跑,读指针(你的代码)在后面追——只要别被套圈,数据就一个不丢。

## 动手实验

### 第 1 步:让麦克风开口

新建 PlatformIO 工程(`board = m5stack-stamps3`,依赖 `M5Cardputer` 库)。有个坑要先踩平:**M5Unified 里 Mic 和 Speaker 不能同时启用**,必须先 `end()` 一个再 `begin()` 另一个——记住这个约束,挑战任务里录音/放音切换全靠它。

```cpp
#include <M5Cardputer.h>

static constexpr size_t REC_LEN = 240;   // 一次采 240 个点:屏幕每列正好一个
static int16_t recBuf[REC_LEN];          // 库吐给我们的已经是 16-bit PCM 了
M5Canvas canvas(&M5Cardputer.Display);   // 离屏缓冲,第 2 课的老朋友

void setup() {
  auto cfg = M5.config();
  M5Cardputer.begin(cfg);
  canvas.createSprite(240, 135);
  M5Cardputer.Speaker.end();   // 关键:先关扬声器
  M5Cardputer.Mic.begin();     // 再开麦克风(引脚配置库已封装好)
}

void loop() {
  M5Cardputer.update();
  // record() 是异步的:把缓冲交给 I2S+DMA 后台填充,排队成功就立即返回 true
  // (队列满时才会等待)。240 点在 16kHz 下约 15ms——你画的是"刚过去"的
  // 15ms 声音,这点延迟眼睛毫无察觉;若想确认录完,可查 Mic.isRecording()
  if (M5Cardputer.Mic.record(recBuf, REC_LEN, 16000)) {
    canvas.fillSprite(BLACK);
    canvas.drawFastHLine(0, 67, 240, DARKGREY);   // 零电平参考线
    for (int x = 1; x < 240; x++) {
      // 16-bit 样本范围 -32768~32767,映射到中线上下 ±60 像素
      int y0 = 67 - recBuf[x - 1] * 60 / 32768;
      int y1 = 67 - recBuf[x]     * 60 / 32768;
      canvas.drawLine(x - 1, y0, x, y1, GREEN);
    }
    canvas.pushSprite(0, 0);   // 整帧 blit,不闪烁
  }
}
```

刷进去,对着它说话——绿色波形随你的声音跳动。**为什么代码里完全没出现 PDM?** 因为库 + ESP32-S3 硬件把"PDM 流 → 抽取滤波 → PCM"整条链路都封装了,你在 `recBuf` 里拿到的直接是干净的 16-bit 样本。知道水面下发生了什么,和只会调 API,是两种境界。

### 第 2 步:看见纯净的正弦波

对着它**吹口哨**。说话的波形毛糙杂乱(人声是一堆谐波的叠加),口哨却接近单一频率——你会看到教科书般的正弦波浮现。数一数:屏幕 240 点 ÷ 16000 Hz = 15 ms 宽;如果一屏里有 15 个完整周期,你的口哨就是约 1000 Hz。恭喜,你已经在用自制仪器做测量了。

### 第 3 步:感受幅度

把手机放歌凑近麦克风,音量从小调到大,观察波形什么时候**削顶**(波峰变平)——那就是超出 16-bit 量程的削波失真(clipping),所有录音师的噩梦,你现在亲眼见到了它的样子。

## 挑战任务

**一台入门示波器要上千块,变声器 App 全是广告——今天你用口袋里这台小机器把两样全部白嫖**:平时实时画声波;按 `R` 录 3 秒;按 `1`/`2`/`3` 分别以 0.5×/1×/2× 采样率回放——恶魔低音、原声、花栗鼠音。

只给思路,代码你自己写:

**里程碑 1:状态机。** 三个状态:`SCOPE`(实时波形)、`RECORDING`(录音中,画进度条)、`PLAYING`(回放中)。第 3、4 课的键盘扫描和状态机在这里复用——按键只负责触发状态迁移。

**里程碑 2:录 3 秒。** 16000 Hz × 3 s = 48000 个 int16 = 96 KB,SRAM 放得下,`static int16_t tape[48000]` 即可。别指望一次 `record()` 搞定:分块录,每块几千点,循环填满大缓冲,顺便更新进度条。

**里程碑 3:把 48000 个点画进 240 像素。** 这是本课点题之处:每列对应 200 个样本,只画第一个点会丢掉尖峰,正确做法是**每列取这 200 个样本的 min 和 max,画一条竖线**——这就是所有音频软件里波形缩略图的画法(峰值包络)。录完后把整段"磁带"的全貌画出来。

**里程碑 4:变速回放。** 切换到播放前记得 `Mic.end()` + `Speaker.begin()`。回放用 `M5Cardputer.Speaker.playRaw(tape, 48000, rate)`——同一段数据,`rate` 传 8000 就是恶魔音,32000 就是花栗鼠。想明白为什么**变速必然变调**:样本没变,只是"播放磁带的马达"转快了,所有频率等比例升高,时长等比例缩短。这正是磁带机时代变速效应的数字复刻。

**验收标准:** 波形实时流畅不闪烁;口哨能看到稳定正弦;录音有进度反馈;三档回放音调差异明显;录/放切换不崩溃(想想 Mic/Speaker 互斥)。

**隐藏成就【共振猎人】:** 翻出第 5 课的钢琴固件思路,让 Cardputer 边放一个持续音、边……等等,Mic 和 Speaker 不能同时开?那就用两台设备,或者让扬声器对着另一台的麦克风。观察示波器上的波形如何被"喂"回来越滚越大——这就是 KTV 啸叫(声反馈自激)的完整闭环:麦克风→功放→扬声器→空气→麦克风。解释:为什么啸叫总是锁定在某个特定频率上?

## 深入一层

1. **手写抽取滤波。** ESP32-S3 帮你做了 PDM→PCM,但你可以反向模拟:把 16 kHz 录音每 4 个样本平均成 1 个(4 kHz),再回放。听听损失了什么——高频先消失,这就是奈奎斯特定理的听觉版。
2. **加触发(trigger)。** 现在口哨波形会左右漂移,因为每帧起点相位随机。真示波器的做法:在缓冲里找一个"上升沿过零点"作为绘制起点,波形就会稳稳钉在屏幕上。实现它,你就理解了示波器面板上 Trigger 旋钮的全部意义。
3. **变速不变调可能吗?** 磁带式变速把音调和时长绑死了。查一查 granular synthesis 或 SOLA 算法的思路(把声音切成小颗粒、重叠拼接),不要求实现,想清楚它为什么能解耦两者。

## 检查点

1. PDM 流里只有 0 和 1,幅度信息藏在哪里?接收端靠什么把它还原成 16-bit 样本?
2. 为什么 PDM 时钟必须是 MHz 级,而最终 PCM 采样率只要 16 kHz?"多出来"的采样换到了什么?
3. 麦克风侧用 PDM、功放侧用 I2S,各自省掉了谁的复杂度?
4. 48000 个采样点画进 240 像素宽的屏幕,为什么"每 200 个取第 1 个"是错的,min/max 包络是对的?
5. 用 32 kHz 回放 16 kHz 录的素材,音调和时长各发生了什么?为什么两者必然同时变?
