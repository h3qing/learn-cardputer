## 硬件原理

### GPIO:芯片伸出来的一根"可编程的手指"

GPIO(General Purpose Input/Output)是 MCU 最原始的对外接口:一根引脚,软件可以决定它是"嘴"还是"耳朵"。

- **输出模式**:芯片主动把引脚拉到高电平(3.3V)或低电平(0V),像开关灯。
- **输入模式**:芯片测量引脚上的电压,读回 1 或 0。

输入模式有个新手必踩的坑:**悬空(floating)**。一根什么都不接的输入引脚,就像一根伸在空气里的天线,读到的值随温度、手指靠近、隔壁 WiFi 随机漂移。解决办法是给它一个"默认立场"——**上拉电阻**把它默认拽到高电平,**下拉电阻**默认拽到低电平。ESP32-S3 的每个 GPIO 内部都自带可配置的弱上拉/下拉,一行 `pinMode(pin, INPUT_PULLUP)` 就能开启。记住这个套路:**上拉 + 按键接地 = 没按是 1,按下是 0**。这是嵌入式世界最常见的按键接法,Cardputer 的键盘矩阵也是这个逻辑(低电平有效)。

### 键盘矩阵:56 个键的引脚经济学

如果每个键占一根 GPIO,56 个键要 56 根引脚——StampS3 整个模块都没这么多脚。矩阵的思路是把键排成网格,键装在行线与列线的**交叉点**上:

```
        列0   列1   列2   列3 ...
行0 ────┬─────┬─────┬─────┬────
        [K]   [K]   [K]   [K]
行1 ────┼─────┼─────┼─────┼────
        [K]   [K]   [K]   [K]
行2 ────┼─────┼─────┼─────┼────
        [K]   [K]   [K]   [K]
```

按下一个键 = 把它所在的行线和列线**接通**。扫描流程像点名:

1. 把第 0 行拉低,其余行置为"不激活"(直连 GPIO 时通常设成高阻悬空;Cardputer 用的译码器则是保持高电平,见下节);
2. 逐个读列线(带上拉):哪一列读到低电平,说明"第 0 行 × 那一列"的键被按下;
3. 换第 1 行拉低,重复;扫完所有行为一帧,几毫秒内跑完一轮。

8 行 × 7 列 = 56 个键,只要 8 + 7 = 15 根引脚。但 Cardputer 还嫌多——

### 74HC138:3 根引脚指挥 8 行

74HC138 是一颗经典的 **3-8 译码器**:输入 3 位二进制数(000~111),它把对应编号的那一路输出拉低,其余 7 路保持高。相当于一个"8 选 1 的接线员"——你报房间号,它只给那个房间打铃。

```
  GPIO ×3 ──> ┌─────────┐──> 行0
   (A2 A1 A0) │ 74HC138 │──> 行1
   如 0 1 1   │  3-8    │──> ...
              │  译码器  │──> 行7   (仅编号=011 的行3输出低电平)
              └─────────┘
```

于是驱动 8 行只需 3 根 GPIO:3(选行)+ 7(读列)= **10 根引脚搞定 56 个键**,比直连矩阵又省 5 根。代价是行选通永远"一次只有一行",这恰好就是矩阵扫描需要的行为——硬件天生配合算法。具体用了哪几个 GPIO,别背,查官方原理图(见参考资料)。

### 抖动与鬼键:矩阵的两个"物理 bug"

**抖动(bounce)**:机械触点闭合的瞬间不是干脆的一下,金属片会弹跳几百微秒到几毫秒,电平疯狂横跳 010101。芯片扫描快到能看清每一跳,不处理就会把一次按键读成好几次。**消抖**的思路:状态变化后,必须**稳定保持一段时间(常用 5~20ms)**才算数。

**鬼键(ghosting)**:按住三个恰好构成 L 形的键——(行1,列1)、(行1,列2)、(行2,列1)——扫描行2时,低电平会沿"行2 →(键 行2,列1)→ 列1 →(键 行1,列1)→ 行1 →(键 行1,列2)→ 列2"这条走私通道爬到列2,让列2也读到低电平:第四个键 (行2,列2) 明明没人碰,却"被按下"了。高端机械键盘在每个键上串一个**二极管**,让电流只能单向通过,堵死绕路——这就是"全键无冲(NKRO)"广告词背后的真实成本:104 颗二极管。Cardputer 的键盘没有这笔预算,所以多键同按时它在理论上是"有冲"的——本课挑战里你会亲手做实验验证。(有个变量要留意:上面的经典分析假设未选中的行是悬空的,而 74HC138 会把未选中的行主动驱动为高电平,和走私通道"打架",实际现象可能更微妙。)

## 动手实验

### 实验 1:20 行代码的打字机

新建 PlatformIO 工程(`board = m5stack-stamps3`,配置沿用第 2 课),写入:

```cpp
#include <M5Cardputer.h>

void setup() {
  auto cfg = M5.config();
  M5Cardputer.begin(cfg, true);   // 第二个参数 true:启用键盘扫描
  M5Cardputer.Display.setTextSize(2);
  M5Cardputer.Display.setTextScroll(true); // 写满自动上卷,不然几行字就跑出屏幕
  M5Cardputer.Display.print("> ");
}

void loop() {
  M5Cardputer.update();           // 每次循环驱动一轮键盘扫描
  if (M5Cardputer.Keyboard.isChange() &&    // 状态发生了变化
      M5Cardputer.Keyboard.isPressed()) {   // 且当前有键按下
    auto st = M5Cardputer.Keyboard.keysState();
    for (auto c : st.word) {      // word:当前按住的所有可打印字符
      M5Cardputer.Display.print(c);
    }
    if (st.enter) M5Cardputer.Display.println();
  }
}
```

刷进去,打几个字。注意两件事的分工:

- `M5Cardputer.update()` 就是那台"扫描机器":内部操纵 74HC138 选行、读列、维护按键列表。它必须在 `loop()` 里高频调用——你不调它,键盘就是死的。这是**轮询**模型:不是键盘通知你,而是你不停地去问。
- `isChange()` 是库帮你做的**边沿检测**:只在按键集合发生变化的那一帧为真。删掉它试试——每帧都会重复打印按住的键,你会瞬间理解"电平"和"事件"的区别。

### 实验 2:看见抖动的余波

把 `isChange()` 条件去掉,加一个计数器,数一数"按住 A 键一秒"期间 `isPressed()` 为真的帧数,再对比"快速点按 A 十次"时 `isChange()` 触发的次数是否恰好是 20(10 次按下 + 10 次抬起)。如果偶尔多出来几次,那就是抖动漏网的痕迹。多不多、稳不稳,取决于库替你消抖消到什么程度——别猜,去翻 Keyboard 类的源码(见参考资料),用实验数据对照它的实现。而当你以后直接用 `digitalRead()` 读裸按键(比如 Grove 口外接按钮)时,消抖就得自己写:记录状态翻转的时刻,`millis()` 差值超过 10ms 才承认。

### 实验 3:测量输入延迟

在 `isChange()` 触发时用 `micros()` 打时间戳,通过串口输出相邻两次扫描的间隔。想一想:如果 `loop()` 里塞了一个 `delay(100)`,按键延迟最坏会是多少?这个问题在第 4 课做游戏时会变成手感的生死线。

## 挑战任务

**目标:单词雨打字游戏。** 这块 1.14 寸的小屏幕即将下起一场单词暴雨,而你手里的 56 个键是唯一的武器:单词砸到屏幕底部之前把它打爆,得分;手速跟不上,掉命;每消 5 个单词雨势加急。用上第 2 课的 sprite 双缓冲,别让画面闪。

分步里程碑:

1. **静态单词**:屏幕中央显示一个单词,键入正确前缀高亮变色,打完整个单词就换下一个。核心数据结构:目标字符串 + 已匹配位置索引。
2. **下落**:单词有 `y` 坐标,每帧按速度递增;用 `millis()` 差值驱动移动量,不要用 `delay()`(想想实验 3)。
3. **多词并发**:同屏 2~3 个单词,键入时自动匹配"以当前输入开头"的那个。词库用 `const char*` 数组硬编码 30 个词就够。
4. **游戏性**:分数、生命、逐级加速、Game Over 画面。爆炸效果可以偷懒:画几帧扩散的圆圈。
5. **隐藏成就【捉鬼人】**:写一个矩阵观察模式(比如按 Fn 进入),把 `keysState().word` 里所有当前按住的键实时列在屏幕上。然后同时按住三个键,系统地尝试不同组合,直到屏幕上出现**第四个你没按的键**——鬼键现身。注意:构成 L 形的是**电气矩阵**的行列,不是键帽的物理排布,所以要靠实验找(这正是乐趣所在)。找到后向朋友复述电流绕路的路径,讲清为什么加二极管能防住。如果翻遍组合也召唤不出干净的第四个键,恭喜——你可能撞见了 74HC138 推挽输出压制鬼键的现场,把"哪些组合会丢键/多键"的实测结果记下来,这份实验报告的含金量不输捉到鬼。

验收标准:连续游玩 3 分钟无崩溃、无闪屏;按键响应无可感知延迟;能现场演示鬼键并说清成因。

## 深入一层

- **绕开库,裸读矩阵**:对照官方原理图找到 74HC138 的 3 根选择线和 7 根列线,自己用 `digitalWrite()` / `digitalRead()` 写一个最小扫描器,和库的结果对照。这是"我真的懂了"的终极检验。
- **N-key 极限测试**:用矩阵观察模式测出 Cardputer 最多能同时可靠识别几个键?哪些组合两键就冲突?画出你推测的部分电气矩阵图。
- **中断 vs 轮询**:ESP32 的 GPIO 支持电平变化中断(`attachInterrupt`)。想一想:为什么矩阵键盘几乎从不用中断驱动,而单个独立按键常用?(提示:行线在扫描时本身就在不停变化。)

## 检查点

1. 一根配置为输入、什么都没接的 GPIO,读到的值是什么?为什么?怎么修?
2. 不用 74HC138 的话,56 键矩阵最少要几根 GPIO?用了之后是几根?省下的引脚去哪儿了(提示:回看第 1 课的总线地图)?
3. 消抖的时间窗设成 1ms 和设成 200ms 各会出什么问题?
4. 口述鬼键的电流路径:三个键按下时,第四个"鬼"是怎么在扫描中被读出来的?
5. `M5Cardputer.Keyboard.isChange()` 帮你完成了什么?如果没有它,你要自己维护什么数据才能实现同样效果?
