## 本课目标

- [ ] 能解释为什么 `delay()` 是游戏(以及一切实时程序)的天敌,并用 `millis()` 写出非阻塞的固定帧率主循环
- [ ] 能画出"输入 → 逻辑 → 渲染"三段式循环的结构图,并说清为什么三者必须分离
- [ ] 能用一个 `enum` 状态机管理"标题画面 / 游戏中 / Game Over"三个场景的切换
- [ ] 能算出 240×135 全屏刷新在 SPI 上的时间开销,并据此判断帧率上限
- [ ] 交付一个可以塞到朋友手里的完整游戏:有标题、能计分、死了能重开、全程不闪屏

## 硬件原理

这是第一场 Boss 战:没有新芯片、没有新总线。第 2 课你已经会驱动 ST7789V2 屏幕,第 3 课你已经会读键盘矩阵。但"会点亮屏幕 + 会读按键"和"做出一个手感好的游戏"之间,隔着四个软件模式——它们和 SPI、GPIO 一样,是后面每一课都会反复用到的基础设施。

**1. 游戏主循环:一台永动的三冲程发动机**

所有实时程序(游戏、示波器、Web Server)本质上都是同一个循环:

```
        ┌──────────────────────────────────┐
        │  ┌───────┐  ┌───────┐  ┌───────┐ │
   ───► │  │ 输入   │─►│ 逻辑   │─►│ 渲染   │ │ ───► 回到开头
        │  │ Input │  │ Update│  │ Render│ │
        │  └───────┘  └───────┘  └───────┘ │
        │        每一圈 = 一帧(frame)       │
        └──────────────────────────────────┘
```

为什么必须分成三段?想象你把"按一下键就移动一格并立刻画出来"混在一起写:蛇的移动速度会取决于键盘扫描频率,渲染卡顿会拖慢逻辑,而逻辑变复杂又会让按键丢失。分离之后,三段各自只做一件事:输入段把按键状态**快照**下来,逻辑段基于快照更新游戏世界(纯数据,不碰屏幕),渲染段把世界画出来(纯绘图,不改数据)。这就是老师傅说的"数据和表现分离",游戏引擎、React、甚至第 8 课的网络状态机,全是这个思路。

**2. 固定帧率:millis() 是你的节拍器**

`delay(33)` 看似能得到 30fps,但它是**阻塞**的:delay 期间 CPU 完全罢工,不扫键盘、不干任何事。更糟的是实际帧长 = 33ms + 你代码的耗时,蛇越长每帧越慢,游戏速度会漂移。正确做法是用 `millis()`(开机以来的毫秒数)做非阻塞节拍:

```
时间轴 ──────────────────────────────────────►
帧预算:  |←── 33ms ──→|←── 33ms ──→|←── 33ms ──→|
实际工作: [██ 干活 ██..空转..][███ 干活 ███.空转.]
                        ↑ millis() 到点才进入下一帧
```

每圈 loop 先问"距离上一帧满 33ms 了吗?"——没满就直接返回(空转期间还能干别的,比如扫键盘),满了才执行一帧。这样无论逻辑快慢,节奏恒定。这个"到点才干活"的模式,第 7 课红外定时、第 8 课断线重连都会再见到。

**3. 状态机:游戏其实是好几个游戏**

标题画面、游戏中、Game Over,是三个规则完全不同的世界:标题画面里按键=开始,游戏里按键=转向,Game Over 里按键=重开。用一堆 `bool isPlaying、isDead` 互相纠缠很快会失控;正确姿势是一个 `enum` 变量 + `switch`,任何时刻程序**只处于一个状态**,切换只发生在明确的箭头上:

```
   ┌────────┐ 按任意键 ┌────────┐  撞墙/撞自己 ┌───────────┐
   │ TITLE  │ ───────► │ PLAYING│ ──────────► │ GAME_OVER │
   └────────┘          └────────┘             └───────────┘
        ▲                                          │
        └────────────────── 按 R 重开 ◄────────────┘
```

**4. SPI 带宽预算:帧率的物理天花板**

回忆第 2 课:全屏一帧 = 240 × 135 × 2 字节(RGB565)= 64,800 字节 ≈ 518,400 bit。SPI 写时钟按约 40MHz 计(第 2 课的数字,M5GFX 的默认配置,精确值以 [M5GFX 源码](https://github.com/m5stack/M5GFX)为准),光传输就要约 13ms——也就是说哪怕逻辑零耗时,整帧 blit 的上限也就 ~77fps。这就是"带宽预算"思维:先算物理上限,再决定策略。全屏 sprite 推送(第 2 课的双缓冲)每帧固定花 13ms,简单可靠;而"脏矩形"只重画变化的区域——贪吃蛇每帧其实只有蛇头和蛇尾两个格子变了,两个 8×8 格子只需 256 字节,是全屏的 1/250。本课先用全屏 sprite(预算足够),但你要知道这笔账,第 6 课画实时波形时它会决定生死。

## 动手实验

先搭一个"最小游戏骨架":一个方块,用键盘控制它移动,带固定帧率和 FPS 显示。这个骨架就是你挑战任务的地基。

**第 1 步:先感受一下反面教材。** 新建 PlatformIO 工程(board = `m5stack-stamps3`,和第 2 课相同配置,依赖 `M5Cardputer` 库),在 loop 里用 `delay(500)` 移动方块并直接往屏幕画。你会看到两个病症:按键经常没反应(delay 期间键盘扫描停了),画面闪烁(直接画屏,第 2 课讲过的撕裂)。病症记住,下面逐个治。

**第 2 步:非阻塞主循环 + 离屏渲染。** 这是本课核心代码,可直接编译运行:

```cpp
#include <M5Cardputer.h>

M5Canvas canvas(&M5Cardputer.Display);  // 离屏缓冲,第 2 课的老朋友

// ---- 游戏世界:纯数据,不含任何绘图 ----
struct World {
  int x = 120, y = 67;   // 方块位置
  int dx = 2, dy = 0;    // 速度
};
World world;

const uint32_t FRAME_MS = 33;   // 帧预算:33ms ≈ 30fps
uint32_t lastFrame = 0;

void setup() {
  auto cfg = M5.config();
  M5Cardputer.begin(cfg, true);           // 第二个参数 true:启用键盘扫描(第 3 课)
  canvas.createSprite(240, 135);          // 整帧画布放在 SRAM
}

// ---- 输入:只负责把按键翻译成"意图" ----
void readInput(World &w) {
  M5Cardputer.update();                   // 触发一次键盘矩阵扫描(第 3 课)
  // 键帽印着方向箭头的四个键:; = 上  . = 下  , = 左  / = 右
  if (M5Cardputer.Keyboard.isKeyPressed(',')) { w.dx = -2; w.dy = 0; }
  if (M5Cardputer.Keyboard.isKeyPressed('/')) { w.dx =  2; w.dy = 0; }
  if (M5Cardputer.Keyboard.isKeyPressed(';')) { w.dx = 0; w.dy = -2; }
  if (M5Cardputer.Keyboard.isKeyPressed('.')) { w.dx = 0; w.dy =  2; }
}

// ---- 逻辑:只改数据,不碰屏幕 ----
void update(World &w) {
  w.x += w.dx;  w.y += w.dy;
  if (w.x < 0 || w.x > 232) w.dx = -w.dx;  // 撞墙反弹
  if (w.y < 0 || w.y > 127) w.dy = -w.dy;
}

// ---- 渲染:只读数据,画到离屏缓冲再整帧推送 ----
void render(const World &w) {
  canvas.fillSprite(TFT_BLACK);
  canvas.fillRect(w.x, w.y, 8, 8, TFT_GREEN);
  canvas.pushSprite(0, 0);                // 一次 SPI 突发,约 13ms
}

void loop() {
  readInput(world);                       // 输入每圈都读,绝不错过按键
  uint32_t now = millis();
  if (now - lastFrame < FRAME_MS) return; // 没到点,这帧还没开始
  lastFrame = now;
  update(world);
  render(world);
}
```

注意 `readInput` 放在帧节拍**外面**:键盘每圈 loop 都扫(一圈不到 1ms),但逻辑和渲染只在 33ms 节拍上执行。这就是"输入延迟"和"帧率"解耦——按键最多迟 1 帧生效,而不是迟一个 `delay(500)`。

**第 3 步:加 FPS 计数器。** 用一个计数器变量记 render 次数,每当 `millis()` 跨过 1000ms 就把计数存为 fps 并清零,渲染时 `canvas.setCursor(200, 0); canvas.printf("%d", fps);` 画在角落。烧录后你应该看到稳定的 30——试着把 FRAME_MS 改成 16(约 60fps)和 100(10fps),亲手感受帧率和"手感"的关系:10fps 时方块的移动像幻灯片,按键响应也肉眼可见地"黏"。

**第 4 步:加状态机骨架。** 定义 `enum class Scene { TITLE, PLAYING, GAME_OVER };`,在 loop 的逻辑段用 `switch` 分发:TITLE 状态下渲染一行大字"PRESS ANY KEY",检测到 `M5Cardputer.Keyboard.isChange() && M5Cardputer.Keyboard.isPressed()` 就切到 PLAYING。跑通三个状态的空转切换,骨架就完工了。

## 挑战任务

**今晚下班前,你口袋里会多出一台游戏机——贪吃蛇、俄罗斯方块、Flappy Bird 任选其一,做到能直接塞进朋友手里、让他玩到不肯还你。** 第一次做建议贪吃蛇——逻辑最简单,且 240×135 用 8×8 格子能铺出 30×16 的棋盘(占 240×128 像素,底部剩 7 像素余量,正好留给边框或分隔线)。验收标准:游戏可玩、全程不闪屏、按键无迟滞、Game Over 显示分数并能按键重开。

只给思路,代码自己写:

1. **里程碑 1 — 数据建模。** 蛇 = 一串格子坐标(用定长数组 + 头尾下标做环形队列,别用动态内存);食物 = 一个随机格子。先在纸上回答:蛇移动一步,数据上到底发生了什么?(提示:不是所有格子都动,只是头部新增一格、尾部删掉一格——这个洞察同时是脏矩形优化的钥匙。)
2. **里程碑 2 — 让蛇动起来。** 把骨架里的方块换成蛇,方向键改变前进方向(注意:禁止 180° 掉头,想想在输入段还是逻辑段拦截?)。蛇的移动节奏(比如每 150ms 一步)和渲染帧率(30fps)是两个独立的节拍器——用两个 `millis()` 计时器。
3. **里程碑 3 — 吃与死。** 头碰到食物:长一格、加分、食物换位置(注意别刷在蛇身上);头碰到墙或自己:切换到 GAME_OVER 状态。
4. **里程碑 4 — 包装成作品。** 标题画面(游戏名 + 按键提示)、游戏中的分数常驻显示、Game Over 画面(得分 + 最高分 + "R 重开")。重开时记得把 World 整个重置——如果你的初始化散落在各处,这一步会逼你把它收拢成一个 `resetWorld()`,这是状态机给你的架构红利。

**隐藏成就【帧率警察】:** 保留第 3 步的 FPS 计数器,让它在整个游戏里常驻角落。然后做个实验:把渲染改成"每节蛇身单独调用一次 `M5Cardputer.Display.fillRect()` 直接画屏"(不用 canvas),观察蛇变长时 FPS 曲线如何下跌,再换回全屏 sprite 对比。写一段话解释:为什么直接画屏时 FPS 随蛇长下降(N 次小额 SPI 传输,每次都有命令/寻址开销),而全屏 sprite 是恒定的 13ms(一次大突发,开销摊薄)?这笔账算明白,你就真正理解了 SPI 带宽预算。

## 深入一层

- **脏矩形实战:** 基于里程碑 1 的洞察,试着每帧只重画三个格子(新蛇头、被删的蛇尾、食物),不用整帧 sprite。FPS 能到多少?什么情况下这个方案会露馅?(提示:分数文字、Game Over 全屏切换。)
- **输入缓冲:** 快速连按"上、左"想让蛇拐 Z 字,但两次按键落在同一个移动节拍里,第二次会覆盖第一次。用一个小队列缓存转向意图,每步只消费一个——这是街机游戏的经典手感优化。
- **固定时间步进(fixed timestep):** 我们的方案里,如果某帧逻辑+渲染超过 33ms 会怎样?查一下游戏开发经典文章 "Fix Your Timestep!",对比"追帧"和"丢帧"两种策略,想想贪吃蛇适合哪种。

## 检查点

1. `delay(100)` 和"用 `millis()` 等待 100ms"在 CPU 行为上有什么本质区别?为什么后者按键不会丢?
2. 输入、逻辑、渲染三段中,哪一段允许修改游戏世界的数据?哪一段允许调用绘图 API?为什么要这样约束?
3. 240×135 RGB565 全屏一帧多少字节?SPI 40MHz 下传一帧要多久?由此推算纯传输的帧率上限是多少?
4. 你的游戏里有哪几个状态?画出状态迁移图,标出每条箭头的触发条件。
